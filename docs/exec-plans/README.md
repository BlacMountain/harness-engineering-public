# Exec Plans

`docs/exec-plans/` 是项目周期管理区。它记录整理前的草稿输入、当前执行计划、完成记录和技术债，保证任务状态不只存在于聊天记录中。

## 写入条件

- 任务跨多个文件、目录、脚本或 policy。
- 实施前需要明确验收标准、风险、顺序或回滚方式。
- 任务可能中断，需要后续 Agent 接手。

## 目录规则

- `prepare/`：来自外部讨论、临时记录或未整理需求的草稿输入。
- `active/`：进行中的执行计划。
- `completed/`：已完成计划，必须包含结果和验证记录。
- `tech-debt-tracker.md`：完成后遗留但不应阻塞当前交付的问题。

## 执行计划必填内容

- 背景。
- 目标和非目标。
- 验收标准。
- 实施步骤。
- 风险和回滚方式。
- 验证命令。
- 决策日志。

## 迁移规则

- 尚未整理成执行计划的需求草稿先放入 `prepare/`。
- 草稿明确目标、验收、风险和验证方式后，再整理为 `active/` 执行计划。
- 任务开始后，将计划放入 `active/`。
- 任务完成后，将计划移动到 `completed/`，补充完成结果和验证记录。
- 未完成事项进入 `tech-debt-tracker.md`，不得只留在聊天记录中。

## 交付文档工作

当任务产出 API 文档、用户手册、部署说明、SDK 文档、operator docs 或产品说明时，执行计划负责记录工作目标、步骤、风险和验证。交付文档成品归属目标项目团队约定的交付文档位置。

## 禁止内容

- 不把长期产品规格放入执行计划。
- 不把永久架构说明只放入执行计划。
- 不复制外部资料原文。
- 不把 `prepare/` 草稿当作已承诺的实施计划。
